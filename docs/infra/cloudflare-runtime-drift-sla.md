# Cloudflare Runtime Drift SLA — Phase 15

> **GOVERNANCE LAYER ONLY** — This document does not authorize deployment, apply, or Cloudflare mutation.
> All drift items listed are documentation records pending human review.
>
> **Separation notice:**
> - **Live runtime facts** — derived from known runtime context (read-only reference)
> - **Repository intent** — what the repo intends to configure
> - **Terraform ownership** — what IaC manages
> - **Manual review actions** — steps requiring human decision
> - **Unresolved drift** — items not yet reconciled
> - **Evidence outputs** — files generated by scanners or reviews

---

## SLA Classes

All Cloudflare runtime drift items must be assigned one of the following SLA classes at time of detection.

| SLA Class | Definition | Max Age Before Escalation | Approval Required |
|---|---|---|---|
| **Critical** | Live production traffic affected; potential secret exposure; tunnel outage; access policy lockout | 24 hours | Release Approver + Security Reviewer + Review Board quorum |
| **High** | Production hostname mismatch; Worker route overlap; Terraform state divergence > 1 resource | 7 days | DNS Owner or Worker Owner + Terraform Owner |
| **Medium** | Stale config file; orphaned credential reference; non-critical hostname mismatch | 14 days | Cloudflare Runtime Owner |
| **Low** | Documentation-only drift; cosmetic config difference; non-prod item | 30 days | Any owner acknowledgment |
| **Accepted Exception** | Known drift approved for deferral; must be registered in exception register | Until expiry date | Review Board sign-off required |

---

## Aging Buckets

Each drift item is tracked in one of the following aging buckets based on days since detection:

| Bucket | Days Since Detection | Required Action |
|---|---|---|
| **Fresh** | 0–7 days | Assign owner; classify SLA; begin remediation |
| **Aging** | 8–14 days | Weekly status update required; escalate if High or Critical |
| **Stale** | 15–30 days | Board review required; exception register entry if deferring |
| **Overdue** | 31+ days | Mandatory escalation to Release Approver; SLA breach reported in monthly evidence |

---

## Escalation Paths per SLA Class

### Critical
1. Detect → Alert **Release Approver** immediately (within 1 hour)
2. Engage **Security Reviewer** if secret/credential involved
3. Invoke **Phase 13 break-glass** if production traffic is blocked
4. File DR evidence using **Phase 19 restore evidence template**
5. Board review within 24 hours

### High
1. Detect → Assign to **DNS Owner** (DNS drift) or **Worker Owner** (route drift) or **Terraform Owner** (IaC drift)
2. Daily status update in drift report
3. Escalate to Release Approver if unresolved after 5 days
4. File in monthly evidence at 7-day mark

### Medium
1. Detect → Assign to **Cloudflare Runtime Owner**
2. Weekly status update
3. Escalate to board review if unresolved after 14 days
4. Exception register entry if deferring beyond 14 days

### Low
1. Detect → Log in drift report with owner acknowledgment
2. Monthly review at board meeting
3. Auto-escalate to Medium if older than 30 days without closure

### Accepted Exception
1. Exception register entry required (see `docs/infra/cloudflare-drift-exception-register.md`)
2. Risk level must be documented
3. Review Board must sign off
4. Expiry date must be set (maximum 90 days)
5. Monthly evidence must include all active exceptions

---

## SLA Breach Definitions and Consequences

| Breach | Definition | Consequence |
|---|---|---|
| Critical breach | Not resolved or escalated within 24 hours | Mandatory incident report; DR tabletop scheduled |
| High breach | Not resolved within 7 days | Board review; release freeze until resolved |
| Medium breach | Not resolved within 14 days | Auto-escalation to High; exception register entry required |
| Low breach | Not resolved within 30 days | Auto-escalation to Medium |
| Exception expiry breach | Exception not renewed or closed before expiry date | Exception removed; drift reverts to original SLA class |

---

## Drift Closure Requirements

A drift item is considered **closed** when all of the following are met:

- [ ] Root cause identified and documented
- [ ] Remediation action taken (code change, config update, or accepted exception filed)
- [ ] Scanner confirms drift is no longer detected
- [ ] Owner has signed off in drift report
- [ ] Evidence filed in Phase 16 archive (for High/Critical items)
- [ ] Drift report updated with `status: CLOSED` and `closed_date`

---

## SLA Fields for Drift Report Items

Each entry in `docs/infra/cloudflare-drift-report.md` must include these Phase 15 fields:

```
- sla_class: Critical | High | Medium | Low | Accepted Exception
- aging_bucket: Fresh (0-7d) | Aging (8-14d) | Stale (15-30d) | Overdue (31+d)
- detected_date: YYYY-MM-DD
- last_reviewed: YYYY-MM-DD
- exception_id: (if Accepted Exception, reference to exception register)
- owner: <role name>
- status: OPEN | IN_PROGRESS | ESCALATED | CLOSED | ACCEPTED_EXCEPTION
```
